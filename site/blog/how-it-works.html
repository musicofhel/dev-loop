How dev-loop Works

A deep dive into the safety harness for AI coding agents · March 2026

AI coding agents like Claude Code are remarkably capable. They can read your codebase, write new features, fix bugs, run tests, and commit code — all autonomously. But that same autonomy is the risk. An agent can delete your .env file, leak API keys in a commit, run rm -rf /, or push broken code to production. The faster the agent works, the more damage it can do before you notice.

dev-loop is a safety harness that sits between the AI agent and your codebase. It intercepts every action the agent takes, checks it against configurable safety rules, and blocks anything dangerous — all in under 5 milliseconds. For deeper checks like security scanning and test execution, it runs a full gate sequence at commit time.

But it goes further than just blocking bad actions. dev-loop is a closed-loop system: it observes every agent session, scores the quality of its own safety checks, and uses that feedback to optimize both the safety rules and the agent's prompts over time. The result is a system that gets better at protecting you the more you use it.

The Problem

When you give an AI agent access to your terminal and filesystem, you're trusting it not to do anything destructive. That trust works most of the time — but "most of the time" isn't good enough when the agent has sudo access and your production credentials are in a .env file two directories up.

The failure modes are varied:

  • Accidental file destruction — the agent writes to .env, .ssh/config, or *.key files
  • Dangerous commands — rm -rf, git push --force, DROP TABLE, curl | sh
  • Secret leakage — API keys, database credentials, or private keys committed to the repo
  • Runaway sessions — the agent burns through context, tokens, and time without making progress
  • Quality drift — code that passes the agent's review but fails tests, introduces vulnerabilities, or doesn't match the spec

You could review every action manually, but that defeats the purpose of having an autonomous agent. You need something that watches continuously, blocks the dangerous stuff instantly, and catches the subtle stuff at commit time — without slowing the agent down.

The Three-Tier Safety Model

dev-loop uses three tiers of protection, from instant to thorough. Each tier catches a different class of problem at a different point in the workflow.

Tier 1: Real-Time Check Engine (< 5ms)

The first tier runs on every single tool call the agent makes. When the agent tries to write a file, edit a file, or run a shell command, dev-loop's Rust daemon intercepts the call and runs three check modules:

  • Deny List (17 patterns) — Blocks writes to sensitive files like .env, .ssh/*, *.key, *.pem, credentials.*, and anything with "secret" in the name.
  • Dangerous Ops (25 patterns) — Catches risky shell commands: rm -rf, force push, DROP TABLE, curl | sh, chmod 777, sudo rm, and more. Some block outright; others warn.
  • Secret Scanner (15 patterns) — Regex-based detection of API keys (AWS, OpenAI, GitHub PATs, Slack tokens), private key headers, database connection strings with passwords, and high-entropy secret assignments.

The entire check runs in under 5 milliseconds. The agent doesn't even notice the latency. If a check fails, the tool call is blocked before it executes — the dangerous write never touches disk, the destructive command never runs.

How the hooks work

dev-loop integrates with Claude Code via six hooks. PreToolUse fires before Write, Edit, and Bash calls to check for blocked files and dangerous commands. PostToolUse fires after writes to scan content for secrets. Four other hooks manage session lifecycle: SessionStart, SessionEnd, Stop (context guard at 85% usage), and PreCompact (saves state before context trimming). All hooks are fail-open — if the daemon is unavailable, tool calls proceed normally.

Tier 2: Checkpoint Gates (~30 seconds)

The second tier runs at commit time. Five gates execute in sequence, and the first failure blocks the commit:

  1. Sanity — Auto-detects the project's test runner (pytest, jest, cargo test, etc.) and runs the test suite
  2. Semgrep — Static analysis for known vulnerability patterns (SQL injection, XSS, command injection, etc.)
  3. Secrets — Scans the staged diff for leaked credentials (more thorough than Tier 1's per-file scan)
  4. ATDD — Checks that code matches acceptance specs (if the project has them)
  5. Review — Placeholder for human or LLM code review

When all gates pass, a Dev-Loop-Gate: <sha256> trailer is added to the commit message, creating an auditable chain of gated commits. Individual gates can be temporarily disabled with dl kill <gate> when you need to bypass them (escape hatches, not locked gates).

Tier 3: Full Pipeline (On Demand)

The third tier is the complete seven-layer closed loop. You give it a bug report or feature request, and it:

  1. Pulls the issue from the tracker
  2. Creates an isolated git worktree for the agent
  3. Runs the AI agent with scoped permissions
  4. Runs quality gates on the agent's output
  5. Records everything for debugging
  6. If it fails — retries with feedback from the failure
  7. If it succeeds — creates a PR for human review

The Seven-Layer Architecture

dev-loop is organized as seven layers that form a closed loop. This isn't just an organizational metaphor — the output of each layer literally feeds back as input to earlier layers. The system learns from its own behavior.

Layer 1: Intake

The intake layer pulls issues from the project's tracker (dev-loop uses beads, a lightweight issue tracker with dependency resolution). It resolves blocking relationships, assigns priority, and feeds ready-to-work issues to the orchestration layer.

Key commands: br ready shows what's ready to work on, br graph visualizes the dependency graph. The intake layer also tracks DORA metrics (deployment frequency, lead time, failure rate, recovery time) for the team's dashboards.

Layer 2: Orchestration

When an issue is ready, the orchestration layer creates an isolated environment for the agent. This means:

  • Git worktree isolation — each agent works in its own worktree, so there's no shared mutable state between concurrent agents
  • Persona selection — a "bug-fix" persona gets a different model, turn limit, retry budget, and cost ceiling than a "feature" or "refactor" persona
  • Task decomposition — complex issues are broken into subtasks that can be assigned to different personas

The persona system is configurable per-project. A bug fix might use Sonnet with 10 max turns and a $1 cost ceiling, while a feature might use Opus with 25 turns and a $5 ceiling.

Layer 3: Agent Runtime

The runtime layer actually executes the AI agent. It manages:

  • CLAUDE.md scoping — each project gets its own instructions file that the agent reads at session start
  • Persona overlay — additional instructions based on the selected persona
  • MCP tool access — the agent gets access to tools via MCP (Model Context Protocol) servers, one per layer
  • Session NDJSON parsing — the agent's output stream is parsed in real-time for monitoring

Layer 4: Quality Gates

This is the Tier 2 checkpoint system described above, plus additional gates for cost tracking and review. Every gate can be individually bypassed when needed — the philosophy is "escape hatches, not locked gates."

The quality gates layer also maintains a planted-defect corpus: 13 known vulnerabilities (SQL injection, XSS, command injection, path traversal, etc.) that are used to calibrate the gate accuracy over time.

Layer 5: Observability

Every layer emits OpenTelemetry spans. These flow to OpenObserve, where six dashboards provide visibility into:

  • Loop health — span counts, latency, error rates
  • Agent performance — turns, tokens, duration per agent session
  • Quality gate accuracy — pass/fail rates, false positive rates
  • DORA metrics — deployment frequency, lead time, failure rate, recovery time
  • Cost tracking — daily token usage, cumulative monthly cost, top token-consuming runs
  • Calibration health — regression detection across the safety pipeline

The observability layer also supports session replay: every event from an agent session is logged as NDJSON, so you can replay the entire decision-making process when something goes wrong.

Layer 6: Feedback Loop

The feedback loop is what makes dev-loop a loop. When a task fails:

  • The system analyzes the failure reason
  • If it's a gate failure, it retries with feedback about what went wrong
  • If it's a resource limit, it escalates to a human
  • If it detects a pattern (e.g., repeated false positives on a specific rule), it suggests configuration changes

The feedback loop also handles cross-repo cascades: when a change in one project affects a downstream dependency, the system automatically creates issues in the downstream project and kicks off work there.

Human feedback is collected via dl feedback <event-id> correct|false-positive|missed. This feeds into a precision/recall/F1 scoring system that tracks the accuracy of each check type over time. The just calibrate command runs a full regression detection pipeline to make sure safety checks haven't degraded.

Layer 7: LLMOps (Feature-Flagged)

The newest layer uses DSPy and the GEPA (Generate, Evaluate, Propose, Apply) pattern to optimize prompts using real session data. Three programs are currently optimized:

  • Code Review — the LLM-as-judge that scores code changes
  • Retry Prompt — the prompt used when retrying a failed task
  • Persona Selection — the prompt that chooses which agent persona to assign

Training data is exported from real sessions: every code review, every retry, every persona selection becomes a training example. The optimizer compares optimized prompts against CLI baselines using the same evaluation set, ensuring fair comparison.

Why DSPy?

DSPy treats prompts as programs with measurable metrics, not as strings to be hand-tuned. This means the optimization is reproducible, the results are auditable, and improvements compound over time. The GEPA pattern specifically avoids the common pitfall of optimizing on a different dataset than the baseline was evaluated on.

The Rust Daemon

The core of dev-loop's real-time safety is a Rust daemon (dl) that runs in the background. The choice of Rust is deliberate: the daemon needs to intercept every tool call with sub-5ms latency, and it needs to be a single static binary that's easy to install.

Key design decisions:

  • Unix socket communication — the daemon listens on a socket, not a TCP port, for lower latency and no network exposure
  • Fail-open design — if the daemon is down, all tool calls proceed normally. Safety is additive, not a gate.
  • Hot-reload config — dl reload picks up config changes without restarting
  • Append-only event log — every event is logged to a JSONL file with automatic rotation

The binary is 6.5 MB. It handles session management, config resolution (three layers: built-in defaults, global config, per-project config), temporary overrides (dl allow-once ".env" --ttl 600), and live event streaming (dl stream).

Configuration

dev-loop uses a three-layer configuration system:

  1. Built-in defaults — hardcoded in the Rust daemon (the 17 deny patterns, 25 dangerous ops patterns, 15 secret patterns)
  2. Global config — ~/.config/dev-loop/ambient.yaml applies to all projects
  3. Per-project config — .devloop.yaml in the project root

Pattern lists are additive: you can add or remove patterns at each level. This means you can add project-specific allow-patterns (e.g., "allow .env.example writes") without weakening the global deny list. Both global and project configs must have enabled: true for checks to run. 

# .devloop.yaml — per-project config
ambient: true

deny_list:
  remove_patterns: ["*secret*"]       # Too broad for this project
  allow_patterns: [".env.example*"]   # Documentation templates are fine

dangerous_ops:
  allow_patterns:
    - "rm -rf target"                 # Rust build artifacts
    - "rm -rf node_modules"           # Node modules

secrets:
  file_allowlist:
    - "daemon/src/check/secrets.rs"   # Test patterns in source

Tracer Bullets

dev-loop is built using a tracer bullet methodology. Instead of building layers horizontally (finish all of layer 1, then all of layer 2), every feature is implemented as a thin vertical slice that cuts through all seven layers. This ensures every layer works together from day one.

Seven tracer bullets validate every critical workflow:

  1. TB-1: Issue-to-PR — the golden path. Take an issue, assign it to an agent, agent writes code, gates pass, PR created.
  2. TB-2: Failure-to-Retry — proves the loop actually loops. A task fails a gate, the system retries with feedback, the retry succeeds.
  3. TB-3: Security Gate — proves security scanning works in-loop. A planted vulnerability is caught by Semgrep, the agent auto-remediates.
  4. TB-4: Runaway-to-Stop — proves resource control works. An agent exceeds its turn limit, gets stopped, usage is visible in dashboards.
  5. TB-5: Cross-Repo Cascade — proves multi-project support. A change in one repo triggers downstream work in another.
  6. TB-6: Session Replay — proves observability works. A 70-event session is replayed from NDJSON for debugging.
  7. TB-7: LLMOps A/B — proves optimization works. DSPy-optimized prompts are compared against CLI baselines.

Each tracer bullet has a single just tb<N> command. All seven are currently passing. A nightly just calibrate run ensures they stay that way.

Calibration and Feedback Scoring

dev-loop doesn't just run safety checks — it measures how well those checks are working. The calibration system has several components:

  • Conformance tests — 106 test cases (82 pre-tool-use, 24 post-tool-use) that verify the check engine produces the correct verdict for known inputs
  • Planted-defect corpus — 13 real vulnerabilities that the gates must catch (CWE-89 SQL injection, CWE-79 XSS, etc.)
  • Feedback scoring — human labels (correct, false-positive, missed) are aggregated into precision/recall/F1 per check type
  • Baseline comparison — current metrics are compared against a saved baseline, and regressions are flagged
  • History tracking — metrics are appended to a JSONL history file so you can see trends over time

The just calibrate command runs the full pipeline: conformance tests, planted-defect verification, feedback scoring, baseline comparison, and regression detection. It produces a dated report and exits with an error if anything has degraded.

What's Next

dev-loop is a working system — all seven layers are functional, all seven tracer bullets pass, and the safety harness runs in production on real codebases. But there's always more to do:

  • Context compression — intelligent context management for long agent sessions
  • Partial success handling — detecting and recovering from tasks that partially succeed
  • Onboarding automation — just onboard <repo> to add dev-loop to a new project
  • More training data — retry_prompt and persona_select need more session examples before optimization is meaningful

The project is 100% open source under the MIT license. The entire tool stack is free (beyond your Anthropic API key). If you're running AI coding agents and want a safety net that learns from its own mistakes, check out the repo.
